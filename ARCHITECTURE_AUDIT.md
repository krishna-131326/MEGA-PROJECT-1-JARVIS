## Current dependency flow

```text
microphone → SpeechRecognition/Google → infinite loop → processCommand
                                                   ├→ webbrowser
                                                   ├→ musicLibrary → pygame/filesystem
                                                   ├→ requests → NewsAPI
                                                   └→ pyttsx3
```

### FACT

- The command router, business rules, network I/O, OS I/O, and presentation are in one
  function/module.
- Global engine/mixer instances hide dependencies and prevent isolated construction.
- Commands are selected through ordered substring checks.
- `main.py`, `mine.py`, and `gpt_enhanced.py` repeat most imports, globals, routing,
  speech, and wake-loop code.
- There is no interface boundary, dependency injection, domain model, event model,
  persistence boundary, or plugin mechanism.

### RISK

- Any audio/network/browser fault can affect the entire infinite loop.
- Broad exception handling in two entry points masks fault categories and prevents
  deliberate recovery.
- Blocking TTS, recognition, playback, and HTTP make responsiveness unpredictable.
- Substring matching can trigger unintended commands; there is no confirmation policy
  for future side-effecting tools.

## Target architecture

Use a modest hexagonal design; do not start with distributed services.

```text
jarvis/
├── api/             optional HTTP/CLI delivery adapters
├── core/            commands, results, orchestration, policy, configuration
├── llm/
│   ├── protocols.py provider-neutral request/response contracts
│   ├── providers/   xAI and optional local adapters
│   └── router.py    explicit selection/fallback policy
├── voice/           STT/TTS ports and adapters
├── tools/           browser, news, music; schemas and permission metadata
├── memory/          opt-in conversation store and retention policy
├── plugins/         registry only after stable tool contracts exist
├── ui/              CLI first; richer UI later
└── observability/   structured logging, metrics, correlation IDs
tests/
├── unit/
├── contract/
└── integration/
```

Core orchestration must depend on protocols, while concrete adapters depend inward on
those protocols. Model output must never directly execute OS actions: validate tool
arguments, apply allowlists, request confirmation for consequential actions, execute
through a bounded adapter, and record a redacted audit event.

## Architecture recommendation

Start with a modular monolith. A plugin system, workflow engine, vector database, and
multi-agent framework add more operational surface than this repository currently
justifies. Introduce each only in response to a tested use case.
